# VarHash offline pipeline

This folder contains a minimal research pipeline for training a semantic hash
model on variable-size FastCDC chunks cut directly from tar streams.

## Expected inputs

### Chunk manifest

`chunks.jsonl` with one object per chunk:

```json
{
  "sha1": "hex digest",
  "project": "glib",
  "version": "2.31",
  "version_order": 3,
  "tar_path": "glib/glib-2.31.tar",
  "chunk_offset": 1048576,
  "chunk_length": 8192,
  "payload_b64": "base64 encoded chunk bytes"
}
```

### Candidate gain file

`candidate_pairs.jsonl` with one object per query-reference pair:

```json
{
  "query_sha1": "hex digest",
  "ref_sha1": "hex digest",
  "retrieval_gain": 0.42,
  "pair_type": "bootstrap_positive",
  "project": "glib",
  "version": "2.31",
  "timestamp": 20240101
}
```

The gain file can be generated by an external job that evaluates xdelta on
candidate references.

### Optional external bootstrap candidates

`odess_candidates.jsonl` / `finesse_candidates.jsonl` can be provided to
`build_candidate_pairs.py` with one record per candidate edge:

```json
{
  "query_sha1": "query chunk sha1",
  "ref_sha1": "historical reference chunk sha1"
}
```

Extra fields are allowed and ignored except for `pair_type`, which can be used
to override the default source label.

## Scripts

- `build_chunk_manifest.py`
  - traverse `tar_versions/<project>/*.tar*`
  - run FastCDC directly on tar byte streams
  - emit `chunks.jsonl`
- `build_candidate_pairs.py`
  - build historical reference candidates from:
    - exact duplicates
    - Odess top-k
    - Finesse top-k
    - random history negatives
    - recent-version offset neighbors
  - evaluate real `xdelta3` delta size
  - emit `candidate_pairs.jsonl`
- `build_datasets.py`
  - split candidate pairs chronologically by `project + version_order` by default
  - write `exact_duplicate_{split}.jsonl` side files by default
  - keep `head / middle / tail` candidate strata in each query group
  - emit `pair_{split}.jsonl` and `query_groups_{split}.jsonl`
- `train.py`
  - train a byte-level semantic hash model with ranking and hash losses
- `mine_round2_candidates.py`
  - use a trained checkpoint to retrieve model-mined historical candidates
  - emit `model_mined_candidates.jsonl` for round-2 labeling
- `export_hashes.py`
  - export `sha1 -> hash words` for the C++ `varhash` feature

## Requirements

- Python packages:
  - `torch`
  - `fastcdc`
- System binary:
  - `xdelta3`

## Typical workflow

```bash
python learning/varhash/build_chunk_manifest.py \
  --tar-root test_data/tar_versions \
  --output runs/chunks.jsonl

python learning/varhash/build_candidate_pairs.py \
  --chunks runs/chunks.jsonl \
  --output runs/candidate_pairs.jsonl \
  --odess-candidates runs/odess_candidates.jsonl \
  --finesse-candidates runs/finesse_candidates.jsonl \
  --xdelta3 xdelta3

python learning/varhash/build_datasets.py \
  --candidates runs/candidate_pairs.jsonl \
  --output-dir runs/varhash_dataset

python learning/varhash/train.py \
  --chunks runs/chunks.jsonl \
  --query-groups runs/varhash_dataset/query_groups_train.jsonl \
  --checkpoint runs/varhash.pt

python learning/varhash/mine_round2_candidates.py \
  --chunks runs/chunks.jsonl \
  --checkpoint runs/varhash.pt \
  --existing-candidates runs/candidate_pairs.jsonl \
  --output runs/model_mined_candidates.jsonl

python learning/varhash/export_hashes.py \
  --chunks runs/chunks.jsonl \
  --checkpoint runs/varhash.pt \
  --output test_data/varhash.hashes
```

## Notes

- The model input is still tar-stream chunks. We do not unpack tar files for the
  main training path.
- `version_order` is only used to enforce historical candidate pools and
  chronological train/val/test splits. It is not part of the model input.
- `Odess/Finesse` are intentionally not required in the first candidate builder.
  They are optional bootstrap sources and should not be treated as the upper
  bound of the dataset.
- `build_candidate_pairs.py` now supports all five planned candidate families.
  The offset-neighbor pool can be limited to the previous `N` versions with
  `--recent-version-window`.
- `build_datasets.py` defaults to keeping exact duplicates out of the main
  training pairs while preserving them in side files for analysis.
- A practical round-2 workflow is:
  - train on round-1 data
  - run `mine_round2_candidates.py`
  - evaluate real xdelta gain on mined pairs
  - merge mined positives / boundaries / hard negatives back into the next
    `candidate_pairs.jsonl`
